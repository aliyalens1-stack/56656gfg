### Ключевые наблюдения по архитектуре

| Аспект | Состояние | Замечания |
|---|---|---|
| **Двойной backend** (FastAPI + NestJS) | Намеренно, работает | FastAPI = orchestration + Phase E/G/H + compat. NestJS = бизнес-CRUD. Связь — HTTP proxy, не gRPC/queue. |
| **Авторизация** | JWT HS256, общий секрет `JWT_SECRET` | FastAPI генерит JWT при login → NestJS его принимает. NestJS `JwtAuthGuard` + `RolesGuard` живут. **Но**: `verify_admin_token()` в FastAPI требует `role=admin` строго; некоторые admin-эндпоинты в FastAPI используют этот guard, многие — нет. |
| **DB** | MongoDB напрямую из обоих сервисов | Нет очереди (RabbitMQ/Redis Stream), все «события» — синхронные HTTP вызовы FastAPI → NestJS `/realtime/emit`. |
| **WebSocket** | NestJS gateway на `/api/socket.io/`, namespace `/realtime` | ⚠️ Идёт через FastAPI catch-all (httpx). **Polling работает** (логи подтверждают), **upgrade → WebSocket transport не доказан** — httpx не поддерживает WS upgrade. По факту все клиенты на polling. |
| **Запуск NestJS** | subprocess внутри FastAPI | `start_nestjs()` запускает `node dist/main.js` при старте FastAPI. Если NestJS падает — circuit breaker открывается, FastAPI пытается перезапустить. |
| **Realtime emit** | FastAPI делает `POST /api/realtime/emit?event_type=…` → NestJS эмитит всем клиентам | Видно в логах ~раз в 5–10 секунд от Orchestrator. |
| **Фоновые движки** | 4 asyncio task'а в FastAPI startup | Zone Engine (10s), Orchestrator Phase E (10s), Feedback Phase G (15s), Strategy Optimizer Phase H (5min). |

## 2. Проблемы и риски бэкенда

### 🔴 Критические

**B-1. Сломанная compat-алиаз `/api/slots/reserve`**
- `server.py:6298` переписывает `POST /api/slots/reserve` → `POST /api/slots/hold`
- В NestJS контроллер `slots.controller.ts:106` имеет `@Post('slots/reserve')`, но `slots/hold` НЕ существует.
- Тест: `POST /api/slots/reserve` → **404 "Cannot POST /api/slots/hold"**.
- Без compat layer'а catch-all проксировал бы запрос к существующему `slots/reserve`. **Compat намеренно ломает рабочий путь**.
- **Фикс**: удалить функцию `compat_slots_reserve` (server.py:6298–6300) — catch-all сделает остальное.

**B-2. Двойственность `bookings.controller`: route order conflict**
- `bookings.controller.ts` в этом же модуле (`bookings.module.ts`) рядом с `current-job.controller.ts` (без префикса `@Controller()`) и `live-movement.controller.ts` (`@Controller('live')`).
- В Nest порядок регистрации контроллеров может приводить к коллизиям: `@Get(':id')` в BookingsController может перехватить `@Get('provider/current-job')` если CurrentJobController зарегистрирован позже.
- Сейчас работает (мы видели 200 OK), но регрессия при минимальной перетасовке.
- **Рекомендация**: вынести current-job в `@Controller('current-job')` или использовать абсолютные пути.

**B-3. WebSocket upgrade через httpx-прокси**
- Клиенты подключаются к `/api/socket.io/?transport=polling` → **polling работает** через FastAPI catch-all. Но `transport=websocket` upgrade httpx _не_ поддерживает.
- В логах supervisord видим только `transport=polling` запросы. Все клиенты сейчас фактически на long-polling.
- **Симптомы**: повышенная задержка событий, лишний CPU на NestJS (опрос каждые 25 сек).
- **Рекомендация**: или вынести `/api/socket.io/` на отдельный route в Ingress (минуя FastAPI), или поднять reverse-proxy перед FastAPI с поддержкой WS upgrade.

### 🟠 Средние

**B-4. NestJS использует `app.setGlobalPrefix('api')`, но контроллеры в модуле `slots` объявлены с `@Controller('')`**
- `slots.controller.ts:28: @Controller('')` + `@Post('slots/reserve')` → даёт путь `/api/slots/reserve`. Странный паттерн.
- То же в `bookings/current-job.controller.ts: @Controller()` + `@Post('bookings/:id/action/:action')`. Даёт `/api/bookings/...`.
- Работает, но усложняет навигацию по коду — невозможно понять «куда ведёт `bookings/:id/status`» без grep.
- **Рекомендация**: явные `@Controller('slots')` / `@Controller('bookings')` где возможно.

**B-5. `verify_admin_token` JWT-роль расхождение**
- FastAPI выдаёт JWT с полем `role` (строка). NestJS `RolesGuard` ожидает то же поле.
- Но в `auth/login` (server.py:1208) и в seed_data роли — `admin`, `customer`, `provider_owner`, `provider_manager`. Контракт enum в Nest (`UserRole`) должен совпадать. Проверял — совпадает.
- **Тонкое место**: смена ролей в seed без миграции существующих токенов. Не блокер.

**B-6. Дублирующиеся endpoint-семьи (`/admin` vs `/orchestrator` vs `/feedback`)**
- Phase E/G/H в FastAPI выставляет `/api/orchestrator/*`, `/api/feedback/*`, `/api/admin/governance/*`, `/api/admin/strategies` параллельно с NestJS `automation` модулем (`/admin/automation/*`).
- Они **не дублируют** один и тот же ресурс, но логика «авто-управление маркетплейсом» разнесена по двум engine'ам. Риск drift'а конфигурации.
- **Рекомендация**: документировать разделение ответственности.

**B-7. Idempotency / circuit breaker — есть, но без тестов**
- `prod_readiness.py` реализует `idempotency_lookup/commit`, `nest_breaker`, `dispatch_alert`, `write_audit`.
- TTL индексы создаются на старте. Но в `tests/` пусто:

```
$ ls /app/backend/tests
__init__.py
```

- **Рекомендация**: smoke-тесты в `ops/` (`smoke-api-contracts.sh` etc.) — есть, но не в pipeline.

### 🟡 Минор

**B-8. `Mongo _id` в ответах NestJS**
- `/api/admin/users` возвращает `_id` (ObjectId-строка). FastAPI native — тоже `_id`. Контракт по `id` vs `_id` не унифицирован — фронтенд вынужден поддерживать оба.

**B-9. Стартовый seed идемпотентен, но грузит ~1 секунды на каждом перезапуске**
- 8 организаций, 12 услуг, 35 ревью, 6 зон + 288 snapshot'ов и т.д. при каждом старте проверяются. Лучше вынести в одноразовый `ops/db-seed.sh`.
